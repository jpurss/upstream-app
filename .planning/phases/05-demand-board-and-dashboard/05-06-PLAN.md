  **Seed data claim fix (MAJOR — UAT test 13):**
  In `app/(auth)/login/actions.ts`, the `signInAsDemo` function (line 42) claims seed data by transferring ownership from a single placeholder ID that matches the demo role. This means:
  - Admin demo claims from DEMO_ADMIN_ID only (which owns nothing meaningful — just 1 upvote)
  - Consultant demo claims from DEMO_CONSULTANT_ID only (which owns engagements, forks, requests)

  The problem: BOTH demo roles should see the same demand board data. Prompt requests are owned by DEMO_CONSULTANT_ID (ID ...0001) and community users (IDs ...0010 through ...0014). When the consultant logs in, they claim ...0001's requests but the upvotes from community users still reference the original community IDs, which is fine (upvotes show counts, not ownership). But the admin sees NO requests because they only claimed from ...0002 which owns zero requests.

  Fix: Change the claim logic to transfer from BOTH placeholder IDs regardless of role. After the existing transfer block (lines 81-105), replace the single-placeholder approach:

  ```typescript
  // Claim seed data — transfer ownership from BOTH demo placeholders to the new anonymous user.
  // Both demo roles should see the same seed data (demand board, engagements, etc.).
  const DEMO_CONSULTANT_ID = '00000000-0000-0000-0000-000000000001'
  const DEMO_ADMIN_ID = '00000000-0000-0000-0000-000000000002'
  const placeholderIds = [DEMO_CONSULTANT_ID, DEMO_ADMIN_ID]

  for (const placeholderId of placeholderIds) {
    if (user.id === placeholderId) continue

    // Transfer engagements
    await adminClient
      .from('engagements')
      .update({ created_by: user.id })
      .eq('created_by', placeholderId)

    // Transfer forks
    await adminClient
      .from('forked_prompts')
      .update({ forked_by: user.id })
      .eq('forked_by', placeholderId)

    // Transfer prompt requests
    await adminClient
      .from('prompt_requests')
      .update({ requested_by: user.id })
      .eq('requested_by', placeholderId)

    // Transfer upvotes
    await adminClient
      .from('request_upvotes')
      .update({ user_id: user.id })
      .eq('user_id', placeholderId)
  }
  ```

  This replaces the existing single-placeholder block (lines 76-105). The loop iterates both IDs and transfers any data owned by either placeholder to the new session user.

  Delete the old `const placeholderId = role === 'admin' ? DEMO_ADMIN_ID : DEMO_CONSULTANT_ID` line and the single `if (user.id !== placeholderId)` block. The new loop with `continue` guard handles both cases.
