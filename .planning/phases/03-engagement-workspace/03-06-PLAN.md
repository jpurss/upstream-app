  <name>Task 1: Add profile upsert to auth flows and create INSERT RLS policy</name>

  <action>
**Root cause:** The `engagements.created_by` column has `REFERENCES profiles(id)`, but no code path ever creates a profiles row. The admin client must be used because:
1. profiles table has RLS enabled with no INSERT policy for authenticated users
2. The REVOKE statements for the auth hook further restrict access

**In `app/(auth)/login/actions.ts`:**

1. Import `createAdminClient` from `@/lib/supabase/admin`
2. In `signInAsDemo()`, after `signInAnonymously()` succeeds and before `redirect()`:
   - Get the user from the auth response (use `supabase.auth.getUser()` since the session is now established)
   - Call `adminClient.from('profiles').upsert({ id: user.id, name: displayName, role: role }, { onConflict: 'id' })` where displayName is the `display_name` from the metadata (`role === 'admin' ? 'Demo Admin' : 'Demo Consultant'`)
   - The `role` field on profiles should match the demo role parameter
   - Use upsert (not insert) to handle repeated demo logins by the same anonymous user
   - If upsert fails, log the error but do NOT block the login — the profile is best-effort for demo users

3. In `signIn()`, after `signInWithPassword()` succeeds and before `redirect()`:
   - The user already exists from getUser() call on line 19
   - Call `adminClient.from('profiles').upsert({ id: user.id, name: user.email?.split('@')[0] ?? 'User', role: 'consultant' }, { onConflict: 'id', ignoreDuplicates: true })` — this creates the profile on first login if it doesn't exist, and does nothing on subsequent logins
   - Use `ignoreDuplicates: true` so existing profiles (with potentially different roles set by admin) are not overwritten

**In `app/(auth)/signup/actions.ts`:**

1. Import `createAdminClient` from `@/lib/supabase/admin`
2. After `signUp()` succeeds and `data.session` exists (user is immediately confirmed):
   - Call `adminClient.from('profiles').upsert({ id: data.user!.id, name: email.split('@')[0], role: 'consultant' }, { onConflict: 'id' })`
   - Best-effort — don't block redirect on failure

**Create `supabase/migrations/002_profiles_insert_policy.sql`:**

Create a safety-net INSERT policy so that if the admin client approach ever changes, the authenticated user can at least insert their own profile:

```sql
-- Safety net: allow authenticated users to insert their own profile row
-- Primary profile creation uses the admin client (service role) to bypass RLS,
-- but this policy provides a fallback if the approach changes.
CREATE POLICY "profiles_insert_own" ON profiles
  FOR INSERT
  TO authenticated
  WITH CHECK (id = auth.uid());
```

**Important:** The admin client (service role) bypasses RLS entirely, so the INSERT policy is a defense-in-depth measure, not required for the fix to work.
